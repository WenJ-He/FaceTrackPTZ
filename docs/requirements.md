# FaceTrack PTZ 需求说明

## 1. 项目概述

### 1.1 项目名称

FaceTrack PTZ

### 1.2 项目目标

本项目用于从全景摄像头视频流中自动检测人脸，并基于既定 PTZ 控制规则依次驱动海康 PTZ 摄像头执行旋转、变焦和停留；在旋转变焦过程中持续采集阶段性人脸图像，送入基于 Triton Inference Server 部署的人脸识别模型，完成向量提取与向量库检索，并记录随着 PTZ 变焦推进识别效果和相似度是否提升；同时在设备原生网页的视频画面上叠加当前检测到的人脸框和身份信息，供用户实时观察。

### 1.3 背景与现状

已有一套可运行的 PTZ 控制逻辑，具备以下能力：

- 输入全景图中的目标框坐标
- 将目标框换算成海康 PTZ 需要的相对矩形坐标
- 调用海康 SDK 发起 PTZ 旋转与变焦
参考/mnt/e/hwj/hkctl

新项目的重点不是重写底层 PTZ，而是在既有控制能力上增加：

- 全景视频流接入
- 人脸检测
- 目标排序与扫描调度
- 变焦阶段图采样
- 人脸识别与向量检索
- 识别效果提升记录
- 设备原生网页叠加
- 配置化、日志化和可维护的工程结构

### 1.4 版本策略

本项目分阶段推进：

- `MVP`：实现“检测 -> 排序 -> PTZ -> 识别 -> 相似度记录 -> 网页叠加”的闭环
- `V2`：增加简单重试、丢失目标回退、识别稳定性优化、阈值策略优化
- `V3`：考虑更复杂追踪、多帧融合、动态重排序、在线评估看板

本需求文档优先约束 `MVP`，仅描述业务需求、约束和验收标准，不定义代码接口、模块文件名或具体实现方案。

## 2. 核心业务目标

### 2.1 用户目标

程序启动后，用户希望看到以下结果：

1. 登录 `192.168.0.248` 的设备原生网页后，看到带有自定义框和身份标注的视频画面
2. 看到 PTZ 摄像头按既定规则自动旋转、变焦、停留
3. 可以在程序运行后获得某一张脸在“全景 -> 逐步变焦 -> 最终放大”过程中的识别相似度变化报告

说明：

- 本地预览窗口或本地页面不是当前交付必须项
- 若保留本地预览，仅作为开发调试辅助，不作为主输出界面

### 2.2 系统目标

系统应具备以下能力：

- 持续读取全景视频流
- 周期性执行人脸检测
- 启动后优先尽快进入“可检测”状态
- 对检测结果进行过滤，并按从左到右、从上到下的方向排序和队列化
- 对每个人脸依次发起 PTZ 旋转变焦
- 在 PTZ 变焦的多个阶段采集图像
- 将阶段性人脸图送入识别模型
- 将识别向量与数据库中预存的人脸库向量进行比对
- 记录 Top1/TopK 相似度、相似度提升量和最终匹配结果
- 在设备网页上叠加人脸框、身份和关键状态
- 在检测不到人脸时不发起任何 PTZ 指令
- 在人员流动场景下，当前目标完成后继续沿右下方向寻找最近的下一个目标，不回退

### 2.3 关键术语定义

- `检测`：从全景图或 PTZ 图中输出人脸框与检测分数
- `识别`：对人脸图生成特征向量并检索向量库
- `向量库`：预存身份特征向量的数据库或检索层
- `相似度提升`：同一目标在连续变焦过程中，相比初始阶段的 Top1 或目标身份相似度变化
- `识别准确率`：仅在存在真值身份的前提下，判断识别结果是否正确的统计指标
- `快速检测就绪`：程序启动后，检测链路在最短时间内完成 Triton 连通性检查、模型可用性确认和首轮推理预热，进入可稳定输出检测结果的状态
- `扫描游标`：当前轮扫描已经处理到的全景位置锚点，用于保证调度方向持续向右下推进、不回头

说明：

- 若没有人工标注或真值身份，系统只能严格记录“相似度变化”和“候选身份变化”，不能宣称准确率
- 若有真值身份，系统才可以计算 Top1 Accuracy、TopK Accuracy 等识别指标

## 3. MVP 范围

### 3.1 必做功能

- 读取配置文件中的设备地址、视频流地址、Triton 地址、模型名、数据库人脸库参数、停留时长和网页叠加参数
- 打开全景视频流并持续读取帧
- 通过 Triton 的检测模型执行人脸检测
- 程序启动时优先完成检测模型的健康检查与预热
- 过滤掉低置信度和过小的人脸框
- 按“从左到右、从上到下、不回头”的规则对人脸框排序
- 将排序结果依次调度给 PTZ 控制模块
- 按前面已确定的 PTZ 规则执行旋转变焦
- 在每次 PTZ 稳定后采集当前阶段图像
- 对阶段图像中的目标人脸执行识别
- 将识别向量与数据库人脸库进行相似度检索
- 记录每一阶段的识别结果、TopK 相似度和与初始阶段相比的提升量
- 在设备原生网页的视频画面上叠加检测框和身份标注
- 在无人脸时不发起 PTZ 指令
- 在人员流动时，当前目标处理完成后继续向右下方向寻找最近的下一个目标
- 记录关键业务日志和评估日志
- 输出阶段性识别报告，支持 Markdown 报告、结构化日志或两者同时输出

### 3.2 明确不做

以下能力暂不纳入 `MVP`：

- 多摄像头协同调度
- 分布式部署
- 复杂人脸轨迹重识别
- 自动构建或自动维护向量库
- 完整的后台管理平台
- 在线主动学习

### 3.3 MVP 执行逻辑

系统按以下流程运行：

1. 启动程序并加载配置
2. 连接全景视频流
3. 周期性抓取帧并执行人脸检测
4. 若当前帧或当前检测周期无人脸，则不发起 PTZ 指令并继续等待下一次检测
5. 过滤无效框
6. 将有效框按规则排序
7. 基于当前扫描游标选择位于其右下方向上的最近下一个目标
8. 取该目标发起 PTZ 旋转变焦
9. 等待 PTZ 到位稳定
10. 采集当前阶段图像并执行识别
11. 把识别向量与数据库人脸库做相似度比对
12. 记录当前阶段的识别结果
13. 若未达到当前目标的扫描结束条件，则继续下一阶段变焦并重复识别
14. 当前目标完成后，更新扫描游标并继续向右下方向选择下一目标
15. 若当前轮已经处理到最右下且无后继目标，则将扫描游标重置到最左上，开始下一轮扫描

### 3.4 单目标阶段性识别规则

对于每个被扫描的目标，系统至少需要记录以下阶段：

1. `Stage-0`：全景图中的初始检测框
2. `Stage-1...N`：PTZ 逐步旋转变焦后的稳定阶段
3. `Final`：当前目标扫描结束时的最终阶段

每个阶段都必须输出：

- 阶段编号
- 截图时间
- 当前框坐标
- 识别向量
- Top1 身份
- Top1 相似度
- TopK 候选身份及分数
- 相比 `Stage-0` 的相似度变化量

## 4. 输入输出定义

### 4.1 输入

#### 4.1.1 视频输入

- 全景摄像头视频流
- PTZ 画面视频流或截图能力
- 优先支持 `RTSP`
- 输入地址必须通过配置文件指定

#### 4.1.2 设备输入

- 海康 PTZ 摄像头地址
- 端口
- 用户名
- 密码
- 通道号
- 设备原生网页地址

以上信息必须从配置文件读取，不允许硬编码在业务代码中。

#### 4.1.3 模型输入

- 全景检测图像
- 识别阶段人脸图像
- Triton Server 地址
- 检测模型名
- 识别模型名
- 输入张量格式信息

#### 4.1.4 向量库输入

- 数据库连接信息
- 数据库类型
- 人脸库表名或集合名
- 预存身份列表
- 每个身份对应的特征向量
- 检索参数，例如 `top_k`、相似度阈值、索引类型

说明：

- 当前人脸库规模预计约 30 人
- 设计上必须支持后续扩展，不允许把人数规模写死在代码中

#### 4.1.5 评估输入

若需要计算“识别准确率”，必须额外提供：

- 当前目标的真值身份或标注集

若没有真值，仅记录相似度和候选身份变化，不计算准确率。

### 4.2 输出

#### 4.2.1 可选本地调试输出

本地预览不是必须交付项，但若启用调试可视化，至少可显示：

- 当前全景画面
- 所有人脸检测框
- 每个人脸框对应的排序编号
- 当前目标的人脸框高亮显示
- 当前目标的识别身份
- 当前目标的 Top1 相似度
- 当前系统状态，例如 `DETECTING`、`MOVING`、`RECOGNIZING`、`HOLDING`

#### 4.2.2 PTZ 控制输出

系统每次发起 PTZ 控制时，至少要能够记录：

- 当前目标编号
- 当前目标框位置
- 当前目标的排序位置
- 当前处理阶段
- 指令发起时间
- 指令执行结果

#### 4.2.3 识别输出

每次识别至少输出：

- 当前目标编号
- 当前阶段编号
- 当前识别身份
- 当前 Top1 相似度
- 当前 TopK 候选结果
- 是否达到识别阈值

#### 4.2.4 评估输出

系统至少要记录以下评估结果：

- 初始阶段相似度
- 当前阶段相似度
- 相似度变化量
- 最佳阶段相似度
- 最佳阶段身份
- 最终阶段身份
- 最终阶段编号

若存在真值身份，还需输出：

- 真值身份
- Top1 是否正确
- TopK 是否正确

#### 4.2.5 报告输出

系统必须支持至少一种持久化输出方式：

- Markdown 报告
- 结构化日志

推荐同时支持两者。

报告内容至少包括：

- 目标编号
- 扫描开始时间和结束时间
- 每个阶段的身份、相似度和变化量
- 最终匹配结果
- 若存在真值身份，则给出正确与否

#### 4.2.6 网页叠加输出

设备原生网页上的叠加层至少需要显示：

- 当前人脸框
- 人脸对应身份
- 相似度分数
- 当前阶段编号
- 当前系统状态

#### 4.2.7 日志输出

系统至少要记录：

- 程序启动与配置加载结果
- 视频流打开状态
- Triton 连接状态
- 每轮检测的人脸数量
- 每轮排序结果
- 每次发送 PTZ 指令的目标信息
- 每次 PTZ 成功或失败
- 每阶段识别结果
- 每目标的相似度变化曲线
- 扫描游标位置
- 无人脸时的空闲状态记录
- 网页叠加推送状态
- 异常和重试

## 5. 业务规则

### 5.1 人脸过滤规则

默认需要支持以下过滤项：

- `score >= score_threshold`
- `bbox width >= min_face_width`
- `bbox height >= min_face_height`
- 坐标必须在全景图有效范围内

### 5.2 排序与扫描方向规则

`MVP` 固定采用“从左到右、从上到下”的扫描顺序，不要求支持置信度排序模式。

排序与调度规则如下：

- 使用人脸框中心点作为排序参考点
- 全局顺序按“先上后下、同行内先左后右”确定
- 当前轮扫描一旦开始，只允许沿顺序向后推进，不回头处理已在扫描游标之前的目标
- 当前目标处理结束后，应从最新检测结果中选择“位于扫描游标之后”的最近下一个目标
- 若不存在位于扫描游标之后的目标，则视为当前轮已到达最右下
- 当前轮结束后，扫描游标重置到最左上，下一轮重新开始

说明：

- 这里的“最近下一个目标”指在顺序上紧邻当前扫描游标之后的目标
- 若人员流动导致原有队列发生变化，也必须继续沿顺序向后推进，而不是回退到左上已扫描区域

### 5.3 PTZ 控制规则

PTZ 旋转变焦规则继承此前已确定的逻辑：

- 以检测框为输入
- 根据目标框大小和中心位置计算 PTZ 目标区域
- 通过既有 PTZ 控制能力发起旋转变焦
- 每次 PTZ 到位后等待稳定时间
- 同一时刻只处理一个目标

### 5.4 单目标扫描规则

- 每轮扫描以当前检测结果为准
- 一轮中每个目标只处理一次
- 当前目标处理完成后再进入下一个目标
- 当前队列和当前检测结果发生变化时，仍需以扫描游标为界继续向后寻找下一个目标
- 到达最右下后，下一轮才允许回到最左上
- 当前目标内部允许有多个变焦阶段
- 若当前检测结果无人脸，则本轮不发起任何 PTZ 指令

### 5.5 识别触发规则

每次识别触发必须满足：

- 当前目标已完成本阶段 PTZ 调整并稳定
- 当前阶段图像采集成功
- 当前人脸框通过最小尺寸与质量检查

### 5.6 检测优先级规则

系统必须优先保证检测链路的实时性和可用性：

- 程序启动时先检查 Triton 检测模型可用性，再检查识别模型
- 若识别模型暂时不可用，不应阻塞系统进入检测和目标排序流程
- 检测模型预热完成后，系统应尽快进入首轮检测
- 识别链路允许在检测链路稳定后继续初始化

说明：

- `MVP` 阶段，检测链路是系统第一优先级
- 识别链路是扫描质量增强层，但不应拖慢首轮检测启动

### 5.7 无人脸规则

- 若当前检测周期内未检测到人脸，则不得发起 PTZ 指令
- 若正在等待下一目标且重新检测后仍无人脸，则系统保持待机并继续轮询检测
- 无人脸状态应写入日志

### 5.8 相似度记录规则

系统必须记录同一目标在多个阶段中的相似度变化，并至少支持：

- 与 `Stage-0` 的差值
- 与上一个阶段的差值
- 当前目标的最高相似度
- 当前目标的最终相似度

### 5.9 准确率统计规则

必须明确区分两类指标：

- 在线运行指标：相似度变化、Top1 候选身份、TopK 候选身份
- 离线评估指标：Top1 Accuracy、TopK Accuracy、阈值命中率

只有在存在真值身份的前提下，才能统计准确率。

### 5.10 网页叠加规则

- 设备原生网页上的视频画面必须支持自定义框叠加
- 叠加内容至少包括框、身份、分数和状态
- 叠加必须与视频坐标对齐
- 叠加刷新不能明显滞后于识别和 PTZ 状态更新

## 6. 系统约束

### 6.1 总体约束

- 检测、识别、PTZ 控制、识别结果记录、网页叠加应相互独立，不允许强耦合到无法替换
- 检测模型和识别模型统一通过 Triton Inference Server 提供推理能力
- 检测链路优先保证低延迟和快速进入可检测状态
- 扫描方向必须单向推进，且必须通过显式的扫描游标保证不回头
- 评估结果和运行日志都必须能够落盘保存

### 6.2 状态要求

系统运行过程中至少要能区分以下业务状态：

- 启动中
- 连接视频流
- 检测中
- 排序中
- 旋转变焦中
- 识别中
- 结果记录中
- 网页叠加中
- 等待中
- 异常状态

状态变化必须可追踪，并能体现在日志中。

## 7. 配置与数据要求

### 7.1 配置要求

系统必须支持统一配置管理，且至少能够配置以下内容：

- 全景视频流地址
- PTZ 视频流或截图来源
- 设备地址、登录信息、通道信息
- Triton 服务地址
- 检测模型和识别模型名称
- 检测相关阈值
- 识别相关阈值
- 数据库人脸库连接信息
- 扫描停留时长
- 检测周期
- 报告输出方式
- 网页叠加开关和目标页面信息

### 7.2 配置校验要求

程序启动时必须完成配置校验，至少包括：

- 关键配置是否存在
- 数值配置是否合法
- 设备连接信息是否完整
- Triton 服务地址和模型信息是否完整
- 数据库人脸库连接信息是否完整

### 7.3 数据库人脸库要求

数据库人脸库必须满足：

- 能存储身份信息
- 能存储对应的人脸特征向量
- 能支持相似度检索
- 能返回 TopK 候选结果
- 能支撑当前约 30 人的人脸规模
- 后续可扩展到更大身份规模

## 8. 检测、识别与调度要求

### 8.1 检测要求

- 系统应持续进行周期性人脸检测
- 程序启动后应优先让检测链路快速可用
- 检测模型未准备好前，不应进入正常扫描流程
- 识别链路初始化不应阻塞首轮检测启动

### 8.2 调度要求

- 扫描顺序固定为从左到右、从上到下
- 当前轮扫描中，不允许回头处理扫描游标之前的目标
- 当前目标处理完成后，必须基于最新检测结果继续向右下方向寻找最近的下一个目标
- 到达最右下后，下一轮扫描才允许回到最左上
- 若人员流动导致目标变化，也必须继续沿当前方向向后推进

### 8.3 识别要求

- 对每个目标至少要进行初始阶段、变焦阶段和最终阶段的识别
- 每个识别阶段都必须记录身份候选和相似度结果
- 识别结果必须能够与数据库人脸库进行匹配

### 8.4 无人脸要求

- 若当前检测周期无人脸，则不得发起 PTZ 指令
- 无人脸时系统应保持待机并继续轮询检测
- 无人脸状态必须进入日志

## 9. 报告与日志要求

### 9.1 报告要求

系统必须支持输出识别过程报告，至少支持以下一种形式：

- Markdown 报告
- 结构化日志

报告内容至少包括：

- 目标编号
- 扫描开始时间和结束时间
- 各阶段识别身份
- 各阶段相似度
- 相似度变化量
- 最终匹配结果
- 若存在真值身份，则标记是否识别正确

### 9.2 日志要求

系统至少要记录：

- 启动与配置加载结果
- 视频流打开状态
- Triton 可用状态
- 每轮检测到的人脸数量
- 当前扫描顺序和扫描游标位置
- 每次 PTZ 发起与执行结果
- 每阶段识别结果
- 每个目标的相似度变化
- 无人脸状态
- 网页叠加状态
- 异常与重试情况

## 10. 网页叠加要求

### 10.1 功能目标

用户登录设备原生网页后，应能在视频画面上直接看到业务系统叠加的信息。

### 10.2 叠加内容

至少包括：

- 当前人脸框
- 人脸对应身份
- 相似度分数
- 当前处理阶段
- 当前系统状态

### 10.3 效果要求

- 叠加位置必须与视频画面对齐
- 叠加刷新不能明显滞后于识别结果和 PTZ 状态
- 不应影响设备原生视频观看

## 11. 测试与验收要求

### 11.1 重点验证项

至少应验证以下内容：

- 固定扫描顺序是否正确
- 扫描游标是否按规则推进
- 无人脸时是否停止发起 PTZ
- 识别结果是否能正常返回
- 相似度变化是否被正确记录
- 数据库人脸库检索是否有效
- 网页叠加是否可见且位置正确
- 报告是否能正常生成

### 11.2 MVP 验收标准

满足以下条件即视为 `MVP` 完成：

1. 启动程序后，可稳定读取全景流
2. PTZ 能按从左上到右下且不回头的规则依次对人脸进行旋转变焦
3. 当前目标结束后，系统能继续向右下方向处理最近的下一个目标
4. 若全景检测无人脸，系统不会发起 PTZ 指令
5. 每个目标在多个变焦阶段都能触发识别
6. 系统能记录同一目标的相似度变化
7. 系统能输出 Markdown 报告或结构化日志报告
8. 若存在真值身份，可统计基本准确率
9. 设备网页可看到叠加框和身份信息
10. 关键异常不会直接导致进程崩溃

## 12. 非功能性要求

### 12.1 实时性

- 检测链路应优先保证快速就绪
- 检测性能不应被识别链路明显拖慢
- 网页叠加刷新应尽量接近实时

### 12.2 可扩展性

- 后续应支持更换检测模型
- 后续应支持更换识别模型
- 后续应支持扩大数据库人脸库规模
- 后续应支持更复杂的扫描与追踪策略

### 12.3 可观测性

- 运行过程必须可通过日志判断
- 识别评估结果必须可追溯
- 报告输出必须可保存和复查

## 13. 风险与限制

### 13.1 主要风险

- 全景视频流不稳定会影响检测节奏
- Triton 推理延迟可能影响实时性
- 检测模型未预热时首帧检测可能偏慢
- 识别分数提高不一定等于识别正确
- 若没有真值身份，无法严格统计准确率
- 人员流动会导致检测结果变化，若扫描游标处理不当容易出现回头或跳跃
- 网页叠加效果可能受设备网页渲染方式限制

### 13.2 当前限制

- 第一版优先保证闭环稳定，不追求最复杂追踪
- 第一版优先记录相似度变化，再逐步完善准确率评估
- 当前人脸库规模按约 30 人考虑，但实现上不能写死

## 14. 补充强制约束

### 14.6 相似度统计口径

相似度统计口径必须统一。

每个阶段必须同时记录两类指标：

1. 动态 Top1

- 当前阶段 Top1 身份
- 当前阶段 Top1 相似度

2. 固定身份

- 固定身份定义为 `Stage-0` 的 Top1 身份
- 后续所有阶段都必须记录该固定身份的相似度变化

3. 变化量

必须记录：

- 相对 `Stage-0` 的变化量
- 相对上一阶段的变化量

### 14.7 向量库实现约束

`MVP` 必须使用本地向量库的轻量实现。

允许方案：

- 内存向量加余弦相似度计算
- SQLite 加向量字段加内存计算

禁止方案：

- `Milvus`
- `FAISS IVF/PQ`
- 分布式向量库

### 14.8 网页叠加实现约束

`MVP` 验收口径：

系统必须提供一个网页入口，满足以下条件：

- 能显示视频流
- 能叠加检测框、身份和状态
- 叠加内容与视频坐标对齐

允许实现方式：

- 本地 Web 页面，例如基于 `Flask` 或 `FastAPI` 提供页面和前端 overlay

明确说明：

- 不要求修改设备固件
- 不要求真正侵入设备原生网页
- 不要求直接改 RTSP 流

禁止误解：

- 不要求必须在设备官方页面 DOM 内叠加
- 不要求 Hack 海康网页

### 14.9 状态机

必须实现明确状态流转：

- `INIT`
- `CONNECTING_STREAM`
- `DETECTING`
- `SORTING`
- `MOVING`
- `RECOGNIZING`
- `HOLDING`
- `NEXT_TARGET`
- `DETECTING` 循环

异常状态流转必须支持：

- 任意状态进入 `ERROR`
- `ERROR` 后进入 `RECOVER`
- `RECOVER` 后返回 `DETECTING`

### 14.10 异常处理策略

视频流：

- 失败后自动重连
- 重连间隔必须可配置

Triton 检测：

- 失败后跳过本轮检测

Triton 识别：

- 失败后记录失败
- 不阻断主流程

PTZ：

- 失败后重试 `N` 次
- 超过重试次数后跳过当前目标

数据库：

- 失败后标记识别不可用

Overlay：

- 失败后不影响主流程

### 14.11 默认参数

系统必须提供以下默认值：

- `row_bucket: 80`
- `max_zoom_stages: 5`
- `recognition_accept_threshold: 0.6`
- `min_similarity_gain: 0.02`
- `ptz_retry: 3`
- `detect_interval_ms: 500`
- `ptz_stable_wait_ms: 1500`
- `top_k: 5`

### 14.12 日志强制要求

系统必须使用结构化日志，格式为 JSON。

每条日志必须包含以下字段：

- `timestamp`
- `state`
- `target_id`
- `stage`
- `event`
- `result`

### 14.13 报告强制结构

每个目标都必须输出如下结构的报告内容：

```md
- Start Time:
- End Time:

### Stage 0
- identity:
- similarity:

### Stage 1
...

...

- best similarity:
- final identity:
- gain:
```
